# Task Manager API

Управление списком задач на FastAPI + SQLAlchemy + SQLite.
## Требования

Python 3.11+ и установленный uv.

## Запуск

```bash
uv sync
uv run uvicorn main:app --reload
```

Swagger: http://127.0.0.1:8000/docs

Чтобы авторизоваться в Swagger - нажать кнопку "Authorize" вверху, ввести username и password.

## Эндпоинты

### Пользователи

- `POST /users/register` - регистрация (JSON: username, password)
- `POST /users/login` - логин через OAuth2-форму, возвращает JWT-токен

### Задачи (все требуют токен)

- `POST /tasks/` - создать задачу
- `GET /tasks/?sort_by=priority` - список задач с сортировкой (title, status, priority, created_at)
- `GET /tasks/search?query=python` - поиск по подстроке в заголовке/описании
- `GET /tasks/top?n=5` - топ-N по приоритету
- `GET /tasks/{id}` - одна задача
- `PUT /tasks/{id}` - обновить (можно передать только нужные поля)
- `DELETE /tasks/{id}` - удалить

## Валидация

- Статус задачи: только "в ожидании", "в работе", "завершено" (enum)
- Приоритет: целое число от 1 до 5
- Каждый пользователь видит и редактирует только свои задачи

## Кэширование

Кэшируется `GET /tasks/top` - этот запрос включает сортировку + LIMIT,
вызывается часто с одними и теми же параметрами, а данные меняются реже чем читаются.

Реализация: словарь в памяти, ключ `(user_id, n)`.
Сбрасывается при создании, обновлении или удалении задачи.

Ограничения: не переживает рестарт, не разделяется между процессами.

## Структура

```
main.py       - точка входа, lifespan
database.py   - подключение к SQLite через SQLAlchemy
models.py     - модели User и Task
schemas.py    - Pydantic-схемы с валидацией

routers/
  users.py    - регистрация и логин
  tasks.py    - CRUD, сортировка, поиск, топ-N
```

## Нагрузочное тестирование

Для нагрузочного тестирования используется Locust

### Проверяемые сценарии

Locust имитирует пользователей, которые:

1. регистрируются через `POST /users/register`
2. логинятся через `POST /users/login`
3. создают стартовые задачи;
4. массово создают новые задачи через `POST /tasks/`
5. получают список задач через `GET /tasks/`
6. многократно вызывают кэшируемый endpoint `GET /tasks/top?n=5`

### Анализ результата

Сравнивались `POST /tasks/`, `GET /tasks/` и `GET /tasks/top?n=5`.

`POST /tasks/` проверяет массовое создание задач и дополнительно сбрасывает кэш пользователя
`GET /tasks/` используется как обычный read-запрос к списку задач  
`GET /tasks/top?n=5` — кэшируемый endpoint: повторные запросы одного пользователя могут возвращаться из `_top_cache` по ключу `(user_id, n)`

Для оценки смотрелись количество ошибок, RPS, median, average и p95 response time. 
По результатам Locust-прогона за 57 секунд было выполнено 3879 запросов, ошибок не было. Кэшируемый endpoint `GET /tasks/top?n=5` показал более низкое время ответа, чем обычный `GET /tasks/`: median 3 ms против 4 ms, average 3.97 ms против 5.23 ms, p95 7 ms против 12 ms. Это соответствует ожидаемому эффекту кэширования: повторные top-N запросы обрабатываются стабильнее и быстрее, пока кэш пользователя не сброшен созданием новой задачи.

### Запуск Locust с HTML-отчётом

1. Запуск приложения `uv run uvicorn main:app --host 127.0.0.1 --port 8000`
2. Запуск Locust с HTML-отчётом 
```
uv run locust -f locustfile.py \
  --host http://127.0.0.1:8000 \
  --headless \
  --users 20 \
  --spawn-rate 5 \
  --run-time 1m \
  --html locust-report.html
  ``` 

  HTML-отчёт Locust: `locust-report.html`
